## Why We're Here

**LogoMesh** is a GitHub App that attacks pull requests with adversarial inputs before production does. It generates fuzz tests for every changed function, runs them in a sandbox, and only posts a comment when it has a concrete, reproducible crash. The core attack engine comes from the Berkeley RDI AgentBeats competition — LogoMesh placed **1st out of 1300+ teams** in Phase 1 of that competition, competing in the **Software Testing track**.

**This repo (`logomesh-lambda`) is not the Phase 1 repo.** It is the Lambda Custom Track repo — a separate AgentBeats track focused on adversarial attacker vs. defender LLM scenarios (prompt injection, jailbreaking, information disclosure). We are adapting our Phase 1 attack engine to run inside this arena's sandbox so we can observe what happens to the attacker and defender models during a scenario: their hidden states, router logits, and internal telemetry. That observation capability is the foundation of V3.

**Submissions are commit-based, not PRs.** You push to your own private repo (created from the Lambda template); commits with `[submit]` keywords trigger GitHub Actions that upload the agent. This means the code in this repo must remain working and competition-legal at all times. V3 changes are additive only — the existing attacker and its 4-call live loop must continue to work as the fallback. If a V3 change breaks the attacker, the submission fails and we lose standing. Never remove or restructure existing competition infrastructure; only extend it.

**The research foundation** is the LaTeX proposal at `temp-docs/03.20.2026 Lambda Custom Track/Lambda_Telemetry_Proposal_V3.tex` — *"Operationalizing Micro-Telemetry to Bound Contextual Debt"*. Read this to understand *why* the architecture works. The engineering translation — *what to build and how* — is `lambda-docs/LAMBDA_V3_ACTION_PLAN.md`. Use both together; the action plan without the .tex is just a list of files to create.

The core argument:

> The Lambda competition limits attackers to **4 LLM API calls per battle**, making live MCTS mathematically impossible. The solution is to **invert the constraint**: run MCTS offline using local open-weight models with H-Neuron and LAT telemetry as the reward signal, pre-compute thousands of adversarial payloads, and deliver the best ones live using 1–2 calls instead of 4.

Two telemetry signals guide offline payload quality scoring — measured on the **defender simulacrum**, not the attacker:
- **H-Neurons** — sparse feed-forward neurons whose activation spikes when a model is being coerced into over-compliance. High activation on the defender = attack is working.
- **LAT probes (RepE)** — linear probes on the residual stream that decode the model's internal "honesty" state. Low honesty projection on the defender = the attack is degrading the defender's integrity = a good payload.

The defender simulacrum is built by loading TinyLlama (Phase A) or gpt-oss-20b (Phase B) with the **exact system prompt the real defender uses** (from `system.j2` + scenario plugin data). Attacks are fired at this simulacrum; the simulacrum's internal signals are the reward. `check_success()` in each scenario plugin provides ground-truth binary labels (did the attack actually work?).

**Dual-use:** The same offline run that builds the offensive payload library also produces a defender vulnerability report — which attack strategies/mutations most reliably coerce the defender, used to harden `defender/agent.py`.

**Secondary goal:** The dataset generated by this process is targeted at the NeurIPS 2026 Datasets and Benchmarks track.

### How Battles Work

Each battle is a multi-round conversation (up to 7 rounds) between an attacker and a defender in a specific scenario. The attacker knows the full attack objective and remembers all prior rounds. The defender is **stateless** — fresh context each round, no memory of prior rounds. Attack succeeds in any round → battle ends early. Defender survives all rounds → defender wins.

**Two attack types:**
- **Direct** — attacker sends messages directly to the defender (Thingularity, GymJailbreak, Medical Records)
- **Injection** — attacker's content is embedded in data the defender processes without direct contact (PortfolioIQ, Resume Downgrade)

**What the attacker sees:** full scenario context (attack goal, constraints, defender role + task, output format, input template), full chat history, `self.round_data` dict for cross-round storage.

**What the defender sees:** filtered context only (role, task, output format, formatted input). No attack info. No history.

**Defender gate:** defenders must pass a normal user helpfulness test before entering battles. Fail it and you cannot compete. A defender that refuses everything is useless.

- Leaderboard refreshes once per day.

## Hard Rules

1. **No hardcoding** scenario-specific logic — competition anti-cheat. Payload libraries are generated dynamically by MCTS, not hand-authored.
2. **4-call budget** per round in live battles — hard limit enforced by `_calls_remaining`.
3. **Teammates submit**, not Josh. Josh does not run live battles with personal keys.
4. **Always apply V3 changes to both sage directories** (they are maintained separately):
   - `scenarios/security_arena/sage/` ← **canonical** (used at runtime + all tests)
   - `scenarios/security_arena/agents/attacker/sage/` ← copy with different defaults
5. **Run tests before committing**: `uv run pytest tests/ -v` — must be 198/198.
6. **Don't touch the 4-call live attacker loop** — it is the fallback. Only extend it, never replace.

## Gotchas

- **Two `sage/` directories.** `security_arena/sage/` is canonical. The test runner adds `scenarios/security_arena` to sys.path first, so this is what gets imported. If you only edit `attacker/sage/`, the change won't take effect at runtime.
- **gpt-oss-20b is MoE (Mixture of Experts).** 32 experts, 4 active per token. Do NOT use dense MLP neuron monitoring for H-Neurons — use **router logit entropy** instead (`output_router_logits=True`). Dense monitoring applies only to Llama-3.2-1B (Phase A).
- **`whitebox.py` already has `RepresentationEngineeringProbe`** — the LAT probe skeleton is built. Implement `_extract_refusal_direction()` in `whitebox.py` in Phase A, don't create a new file.
- **transformers dev required:** `GptOssForCausalLM` is not in the stable release. Install with `pip install git+https://github.com/huggingface/transformers`.
- **MXFP4 quantization:** gpt-oss-20b weights are already 4-bit. When loaded via transformers, computation runs in bfloat16. Adding another quantization layer (GGUF, GPTQ) is double-quantization — don't do it.
- **RTX 3060 12GB limit:** gpt-oss-20b (~13–15GB inference) exceeds Josh's local VRAM. Phase A uses Llama-3.2-1B only. Phase B runs on Lambda H100.
- **`AblationConfig.enable_self_eval` defaults differ between sage copies.** The canonical `security_arena/sage/ablation.py` has `True` (research defaults). The `attacker/sage/ablation.py` has `False` (optimized for gpt-oss-20b — wastes a call). This is intentional.
