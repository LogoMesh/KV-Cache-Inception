## 1. E&D Track Data Requirements

### Submission requirements

✅ **Data is a primary deliverable** — must be submitted by May 6, 2026 alongside the paper.
It is not supplementary.

✅ **Reviewer accessibility** — dataset must be instantly accessible to reviewers with no
human mediation. No "email the PI" gates, no institutional approval workflows during review.

✅ **Approved hosting platforms:**
- Harvard Dataverse
- Kaggle
- HuggingFace
- OpenML
- Bespoke / self-hosted (only if dataset technically requires it)

✅ **Large dataset handling:** For datasets larger than 4GB, authors must provide a
representative subsample and explain the sampling methodology.
(⚠ PARTIALLY VERIFIED — stated in E&D guidance; exact procedure not fully specified in handbook)

✅ **Croissant 1.1 metadata is mandatory** for E&D track dataset submissions.
Must include RAI (Responsible AI) extension fields. Non-compliance is grounds for
desk rejection per the E&D track page.

✅ **Core Croissant structural fields** are auto-generated by HuggingFace, Kaggle, and
Harvard Dataverse when a dataset is uploaded. RAI fields must be manually provided by authors.

### Platform size limits

⚠ **UNVERIFIED** — The AI-generated guides claimed:
- HuggingFace: 300GB
- Harvard Dataverse: 1TB (2.5GB per file)
- Kaggle: 200GB
- OpenML: 200GB

These figures do not appear on the official NeurIPS E&D page. Verify directly against each
platform's current documentation before committing to a host. Current MCTS trace data is tiny
(< 1MB); this becomes relevant at H100 scale.

## 3. What Reviewers Will Look For

For the NeurIPS E&D track, reviewers evaluate the dataset artifact as a first-class contribution.
Based on the handbook, expect scrutiny on:

### Scientific quality
- Does the dataset enable the evaluation claims in the paper?
- Are records diverse enough to support statistical claims?
- Are ground-truth labels present and credibly collected?
- Are the metrics (OEI, TDS, σ_H, ρ_R) correctly implemented per the stated equations?

### Reproducibility
- Can a reviewer reproduce the dataset from scratch given the paper + code + metadata?
- Are seed, git SHA, command, and timestamps present in every artifact? ✅ Yes (our export does this)
- Is the export pipeline runnable from the repo without special access?

### Metadata completeness
- Is Croissant 1.1 JSON-LD present and validator-clean?
- Are RAI fields populated with substantive content (not boilerplate)?
- Is `citeAs` present? (Pending submission ID)

### Access and dual-use posture
- Can reviewers access the data instantly?
- Is there explicit intended-use / non-intended-use language?
- Are high-risk artifacts (intervention traces, attack payloads) appropriately gated?

### Known gaps that reviewers will notice
- No ground-truth labels (honest limitation to state in the paper)
- Small scale (2 runs, 27 nodes) — Phase B will address this
- ρ_R signal uses normalized value instead of raw Eq. 4 dot product — must be fixed before final release

## 4. Hosting Decision

**Team decision required at meeting.** Options:

| Platform | Strengths for our use case | Concerns |
|---|---|---|
| **HuggingFace** | ML community discoverability; dataset cards with Croissant auto-generation for core fields; gating available for dual-use artifacts; reviewer-friendly | Size limit ⚠ UNVERIFIED; anonymization protocol unclear |
| Harvard Dataverse | Institutional permanence; strong DOI + citation support; well-understood by reviewers | Less ML-native; Croissant tooling manual |
| Kaggle | Broad audience | Least relevant ML safety audience; Croissant less native |
| OpenML | Native benchmark/evaluation context; Croissant support | Smaller community; less ML safety focus |

**Tentative recommendation:** HuggingFace gated repository. Rationale: dual-use posture requires
gating (✅ HuggingFace supports this), Croissant auto-generation reduces manual work, and the
ML safety audience for this paper is concentrated on HuggingFace. Anonymization strategy for
the review period is the key open question.

**Before creating the hosting account:**
- Decide platform
- Determine anonymization strategy (anonymous org, private-during-review, etc.)
- Verify actual size limits on the chosen platform
